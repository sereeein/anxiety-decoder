## 0. 一句话定位

**焦虑解码器是面向学生与申请者的反内耗启动器** —— 帮你在"想做又动不了"的瞬间，5 分钟内戳穿模糊感、拿到一个能立刻动手的最小动作。

---

## 1. 问题与洞察

### 1.1 用户在哪一刻被卡住

学生、研究生、求职者每天都会遇到这样的瞬间：知道自己应该改简历 / 写论文 / 投实习 / 联系导师，打开文档看一眼又关掉。被回避的不是任务本身，而是任务背后那一团说不清的"嗡嗡作响"。

待办类工具假设用户已经知道要做什么，只是缺排序；冥想 / CBT 工具让用户"接受"焦虑；日记 / 番茄类工具让用户施压完成。三者都没有触及"知道但启动不了"的这一公里。

### 1.2 核心洞察

焦虑型拖延最骗人的地方，是它让你以为有一个具体的坏结果在等你。但把脑子里所有担心倒出来、戳穿之后，往往发现：

- 一部分是 **真实需要处理的事**
- 一部分是 **灾难化想象**
- 一部分是 **根本说不清的雾**

**大约 70% 的任务回避，背后什么具体的事情都没有。** 产品的价值命题就建立在对这个现象的持续验证上 —— 用户每次使用完，都会积累一份属于自己的"灾难没发生"证据库。

这个洞察既是产品的设计起点，也是它要长期验证的命题（见 §9 风险与验证）。

## 2. 目标用户

### 2.1 核心用户：学生 / 研究生 / 申请者

这一类人群的任务形态和产品命题高度匹配：

- **大块、无明确 deadline 的任务**：论文、毕业设计、找工作、申请文书。任务的"边界"由用户自己定义。
- **高频回避，低外部监督**：没有同事 / 老板盯着进度，回避代价显性化得很慢。
- **任务语言含糊**："投简历"、"准备面试"、"写文献综述"。说出来不算骗自己，但具体下一步是什么并不清楚。
- **强灾难化想象**："投了会被拒"、"面试会表现很差"、"导师会觉得我不行"。心智里"假想结果"占用的带宽远大于"真实的下一步"。

### 2.2 为什么不是"所有焦虑型用户"

服务"所有焦虑用户"是一个看起来更大但实际更小的市场，因为它要求产品同时解决治疗式焦虑、社交焦虑、健康焦虑、灾难化焦虑、表现焦虑等一系列性质完全不同的问题。每一类的解法和工具形态都不一样。

学生 / 申请者这个细分有一个独特的组合：**任务回避 + 高自由度 + 低外部监督 + 强灾难化想象**。这个组合让"把焦虑拆成三堆 + 给一个 5 分钟动作"这套机制对他们的边际有效性最高。给一个用 Headspace 缓解工作焦虑的人塞解码卡，他既不是回避具体任务、也不缺下一步，产品根本没有切入点。

为这个细分人群造工具，比为"所有焦虑用户"造泛工具更可能产生留存。

### 2.3 产品边界

**不是**治疗 App、不是 CBT 工具、不是情绪日记、不是冥想陪伴。
**是**一款服务焦虑型使用者的 **生产力工具**。

## 3. 竞品与差异化

### 3.1 三类邻居产品

| 对比对象 | 它们做什么 | 焦虑解码器做什么 |
|---|---|---|
| 待办类（Todoist / 滴答） | 假设你已知要做什么，帮你排序 | 服务"知道但启动不了"这一公里 |
| 冥想 / CBT（Headspace / Woebot） | 让你"接受"焦虑 | 让你"绕过"焦虑去启动 |
| 日记 / 番茄（Daylio / Forest） | 记录心情或施压完成 | 只承担一件事：帮你开始下一步 |

### 3.2 定位的精确性

定位：**生产力工具，服务焦虑型使用者** —— not therapy, not journaling.

这条边界写出来很短，但它在产品设计的每一个细节里都被反复测试：是否要加冥想模块？不加。是否要做情绪曲线？不做。是否要做日记本？不做。任何一个偏离"帮你启动下一个动作"的功能，都被纳入 V2+ 或者直接砍掉（见 §7）。

**视觉走"极简工具"感**（对标 Linear / Raycast），而不是"温暖 App"感（对标 Headspace / Calm）。这是 §9 风险矩阵里"被误读成治疗 App"的核心缓释手段。

## 4. 核心体验设计

### 4.1 单次解码闭环（一次使用内走完）

```
[落地仪式：随机一句反焦虑短句 1.5s 淡入淡出]
        ↓ 自动过渡（用户也可点击跳过）
[空白页 + 一段温柔文字 + 输入框]
        ↓ 用户写一段（自由文本）
[AI 自动识别：启动仪式 OR 救援模式]
        ↓ AI 追问 1~2 个针对性问题（硬上限 2 轮）
[解码卡生成]
   🟢 真问题 → 主视觉 + 「现在做 5 分钟」主按钮
   🟡 灾难化 → 静默入"事后验证队列"
   ⚪ 雾    → 静默存档
        ↓
[用户可手动修正分类 —— 拖/点将 worry 移到另一堆]
        ↓
[轻邮箱询问（可跳过）]"想让我 3 天后问问你那件事发生了吗？"
        ↓ 用户点主按钮
[等候页：陪伴文案 + 唯一一个「我回来了」按钮]
        ↓ 用户回来
[二次触点：刚才怎么样？emoji + 一句话]
        ↓
[历史卡片列表 + 长期证据数据]
```

### 4.2 长期价值循环（跨会话，异步）

```
解码时 catastrophic 类 worry 被标记
        ↓ 1-7 天后（用户选定）
[邮件触发 OR 站内兜底提醒]"当时担心的事发生了吗？"
        ↓ 用户回答
[单条 worry 标记为 did_happen: true/false]
        ↓ 累积
[个人证据库："你 30 次解码里，22 次灾难化想象最终没发生"]
```

短期循环负责"价值兑现 = 用户离开时确实开始动了"。长期循环负责"复利兑现 = 用户多次使用后形成对自己思维模式的元认知"。两个循环互相支撑：没有短期，长期不会发生；没有长期，短期就只是一次工具触达。

### 4.3 落地仪式（Opening Ritual）

每次用户访问 `/` 时，先呈现一句"反焦虑短句"，1.5s 淡入展示 + 0.5s 过渡，再自动接入输入表单。

**为什么必要**：

焦虑用户从其他页面切过来时，脑子是嗡嗡的；直接看到输入框反而会卡住"该写什么"。一句简短的、温柔的、非治疗式的短句，作为心理"减速带"，让用户从"日常焦虑回路"过渡到"愿意写下来"的状态。

这是把"必然的首屏渲染时间"变成产品价值的微设计 —— 加载页本来就是用户在等的 0.5-2s，把它装进仪式里，等待不再是损耗。

**呈现规范**：

| 维度 | 规范 |
|---|---|
| 时长 | 1.5s 显示 + 0.5s 淡出过渡到输入框，总计约 2s |
| 不阻塞 | 用户点击任意位置可立即跳过 |
| 视觉 | 纯净背景 + 单行短句居中，无装饰、无 logo、无加载圈 |
| 字号字体 | 与解码卡 `card_headline` 同款字体（视觉仪式感的一致性） |
| 减弱动效 | 命中 `prefers-reduced-motion: reduce` 时去掉淡入淡出，直接显示 + 缩短到 1s |

**短句内容原则**：

- ≤ 20 个汉字
- 工具语气，不是疗愈语气：观察句、锚定句优先；劝告句（"你应该"）禁用
- 不出现疗愈话术词汇："焦虑""治愈""放松""平静""加油""勇敢"等禁用
- 留白感：让读者自己接话，不替读者总结

**v1 种子库（10 条）**：

- 不去想两个小时以后和五公里外的事
- 你脑子里 90% 的事此刻都不需要解决
- 现在不知道答案是允许的
- 只决定接下来五分钟
- 焦虑常常是脑子在做没人要求它做的工作
- 想得越远，能动的越少
- 把一件事做完不是仪式，开始才是
- 你不需要先平静下来才能开始
- 脑子比你以为的可信度低
- 下一步比下个月更重要

抽样规则：纯随机，但避免最近 3 次显示过的（用 localStorage 记录最近 3 条 index）；全部用完前不重复。抽样逻辑放在 `/lib/core/quoteSelector.ts`（纯函数，便于单测）。

### 4.4 信息架构

| 路径 | 角色 |
|---|---|
| `/` | Landing — 空白 + 引导文 + 输入框（主入口） |
| `/decode/[sessionId]` | 对话阶段（AI 追问） |
| `/decode/[sessionId]/result` | 解码结果（卡 + 主按钮） |
| `/decode/[sessionId]/wait` | 等候页（陪伴文案 + 「我回来了」） |
| `/decode/[sessionId]/return` | 二次触点（刚才怎么样） |
| `/history` | 历史卡片列表 + 长期证据数据 |
| `/card/[sessionId]` | 单卡详情页（可分享 / 截图） |
| `/verify/[token]` | 邮件点回的事后验证页 |
| `/settings/data` | 数据驾驶舱（查看 / 导出 / 删除） |
| `/privacy` | 隐私说明 |

`/decode/:sessionId/*` 是同一会话的状态推进，用 Next.js nested route 支持深链恢复；`/history`、`/card/:id`、`/verify/:token` 是真正可外链的页面。

### 4.5 Session 状态机

```
draft → conversing → decoded → launched → returned → completed
                                    ↓
                             （有 catastrophic item 的）
                                verification 独立生命周期（异步）
                                    ↓
                              pending → sent → responded
```

## 5. 9 个关键设计决策

### 5.1 单入口取代模式选择

最容易忽略的产品选择：要不要让用户在 landing 上选"启动模式 / 救援模式"？

不分。AI 从首句话自动识别状态，动态调整后续追问的语气。

权衡：让用户选模式看起来给了控制权，对正常用户是友好的。但对焦虑型用户来说，"现在该选哪个模式"本身就是一个需要决策的问题，会触发"选错怎么办"的二阶焦虑，导致大量用户在 landing 卡住根本不进入产品。把识别工作移到 AI 端、让用户的第一个动作只是"写一段话倒出来"，是用工程换体验。

代价：AI 偶尔会识别错状态，整体口吻会有一次轻微的偏移（比如本应该用"救援"语气的，给了"启动"语气）。这个代价远低于选择题导致的入口流失。

### 5.2 AI 架构从 6 次调用演化到 3 次关键路径调用

第一版设计中，提出了"按 A 类（结构化）/ B 类（温度敏感）路由 DeepSeek + Claude 双供应商"的架构。在实现规划阶段对完整关键路径做了延迟分析后发现：6 次调用累计 12-15s，对焦虑型用户而言每多 1s 都在加剧焦虑。

修订后的架构原则：从"按可见性 + 温度敏感性 + 结构化需求三维度路由"，收敛到"按任务风险分级路由 Haiku（轻 / 短 / 可重做）vs Sonnet（高 stakes / 长 stakes / 用户反复看）"。供应商从两家收敛到 Anthropic 一家。

**架构叙事的演化本身比初版方案更值钱**：从"我应用了一个分类模式"升级为"我设计了一个分类方案、测了关键路径延迟、发现它服务故事而非用户、收敛到更简单的版本"。这是工程成熟度的核心证据：有规则 → 测量 → 敢打破规则。

修订后的关键路径 ≤ 12s 累计、单次连续等待 ≤ 5s。

### 5.3 追问硬上限 2 轮

产品原则，不可配置。

为什么硬上限：焦虑型用户对"被审问"极度敏感。AI 的追问越多，用户越觉得自己"说得不够好"或"在被分析"，反而触发新一层的焦虑。两轮足够 AI 拿到分类所需的最小信息；够不够都停。

为什么不可配置：放成参数会让团队在 Phase 4 打磨时反复调"3 轮是不是更准"。一旦开了这个口，产品边界意识就崩了。设计原则比 AI 准确率更重要。

### 5.4 偏心第一堆

解码卡上有三堆 worry（real / catastrophic / fog），但**主视觉焦点只有一个**："接下来要做什么"。其他两堆静默处理。

为什么偏心：用户离开产品时，记忆容量只有一件事。如果三堆同等强调，用户会带走"我有一堆问题"的感觉，价值反而稀释。把全部视觉重量压在 real 那一堆 + 一个 5 分钟动作上，让用户离开时只记得"接下来做什么"，是产品价值兑现最关键的一秒。

catastrophic 和 fog 的价值在长期沉淀里：catastrophic 进事后验证队列形成证据库；fog 进归档形成"我经常在为什么发愁"的元认知。短期不抢戏，长期复利。

### 5.5 允许手动修正分类 + 修正率作为质量反指标

用户可以拖动 / 点选把 worry 从一堆移到另一堆。每次修改写入 `was_manually_edited` 字段。

姿态：**AI 是辅助，不是权威**。给用户最终决定权，避免"AI 硬塞"导致的挫败感。

工程价值：**修正率 = AI 分类质量的反指标**。如果某周修正率 > 20%，说明 prompt 出问题了。这是一个自闭环的质量治理思路 —— 不需要外部标注、不需要自动断言，用户的每一次修正都是免费的训练数据。

阈值设计：MVP 期目标 < 20%。> 20% 触发 prompt 复盘。

### 5.6 「我回来了」必须用户主动点击

不自动检测窗口焦点；让二次触点的触发显式化。

为什么不自动：自动检测会触发"用户切了个 Tab 又切回来"这种伪信号，污染数据；更核心的是，**显式点击本身就是仪式的一部分**。"开始做 5 分钟" → 真去做 → 主动点回来，这条主动闭环本身就是产品价值的载体。

代价：用户可能关掉浏览器忘了回来，导致循环没完成。可接受 —— 这部分流失正是产品要测量的（"主按钮点击率 → 回来率"漏斗）。如果回来率长期偏低，说明陪伴页或主按钮的 action 颗粒度有问题。

### 5.7 邮箱作为 commitment device

解码完成后轻浮现一个邮箱询问："想让我 3 天后问问你那件事发生了吗？"可跳过。

不是为了拉新闻列表订阅，是产品心理学层面的承诺设备：

- 留邮箱 = 对未来的自己多一层问责
- 跳过 = 仍然完成解码，主循环不被打断
- 留了邮箱的用户：3-7 天后收到"那件事发生了吗" → 长期证据库的种子
- 没留邮箱的用户：下次打开时站内兜底"还有 N 条待验证" → 不让 catastrophic 价值流失

邮箱在 MVP 是次级指标（≥15%），在产品成熟期会升级为北极星（详见 §6）。

### 5.8 分类有疑义时倾向「雾」

prompt 中明确的优先级：AI 拿不准时，宁可归入 fog 也不要硬塞 real / catastrophic。

为什么：硬塞 real 会让用户觉得"AI 在编不存在的任务"；硬塞 catastrophic 会让用户觉得"AI 在贴标签说我焦虑"。归入 fog 是最低伤害的兜底 —— 用户最多觉得"AI 不太聪明"，但不会感受到被冒犯或被误解。

这条原则也支撑了产品的核心命题数据收集：fog 占比本身就是"70% 是雾"洞察的直接验证（见 §9）。

### 5.9 落地仪式：把不可避免的加载时间装进产品价值

详见 §4.3。一个不显眼但很值得讲的微设计。

绝大多数 web 产品把首屏加载时间当作"技术不够好的负担"来对待，努力减到最低。焦虑解码器反过来：识别到这 0.5-2s 是必然存在的 → 把它变成产品体验的一部分 → 一句反焦虑短句让用户从"日常嗡嗡的脑子"过渡到"愿意写下来"。

这体现的是对"产品节奏 + 情绪流转"的细颗粒度判断。不是装饰，是把负债转化为资产的微决策。在面试场景里，这是"我会注意到别人不会注意的产品细节"的可演示证据。

### 5.10 验证邮件文案的中性化设计

事后验证邮件是产品里**最高风险的接触点**：发出的瞬间，用户可能正处于"那条担心还没消化完"的状态。措辞偏冷（像审问）会让用户拒答；措辞偏暖（像心理咨询）会重新激活焦虑、违背 §3 "工具不是治疗"的边界。

**3 个文案候选 + 1 个落地版本**：

| 变体 | 调性 | 取舍 |
|---|---|---|
| A · 朋友短信感 | 极简，1 句话 + 链接 | 点击率可能高但价值密度低 |
| B · 工具感 + agency | 强调"加进证据库" | 命题对位但偏理性，丢温度 |
| C · 散文叙事感 | 长，第二段承认"答案是什么都可以" | 情感连接强，但有滑向"治愈系微信公众号"的风险 |
| **E（B+C 中和）** | 保留 B 的 framing + C 的去压力句 | 落地版本 |

变体 E 的关键句：

- 「**那天你写："X"**」（X 是 LLM 提取的 ≤25 字引述，原话直接还给用户）
- 「**N 天过去了。发生了或没发生，说什么都行**」—— 显式去除"我应该答对"的压力（来自 C）
- 「**这一题会进记忆库，是你给那天的自己的一个回应**」—— 把"evidence collection"从工具语言转译到关系语言（来自 B 但柔化）

**LLM 范围收窄**：原版 prompt 让 LLM 写 2 句完整开头（自由度高、漂移风险大）；变体 E 把 LLM 工作切成最小 —— **只生成一个 ≤25 字的引述**，模板拼接其余固定文案。这一改让"LLM 风格漂移"的失败模式被压到几乎为零。

**为什么这是设计决策而不是文案修订**：邮件不是一次性写文案的事 —— 它定义了"产品如何与用户进行非同步对话"的整体姿态。把"中立证据收集"翻译成具体的中文措辞，需要在三种调性之间反复对话才能落地。这份决策的过程比最终的字本身更重要。

## 6. 指标体系

### 6.1 北极星（分阶段）

#### MVP 阶段北极星：正向完成率（Positive Cycle Rate）

**定义**：所有进入二次触点（已点击"我回来了"并填完反馈）的 session 中，情绪反馈为 🙂 的比例。

**为什么是 MVP 阶段的北极星**：

MVP 阶段要验证的核心命题不是"用户会长期依赖这个产品"，而是"**用户单次使用完之后，是否真的感到帮助**"。

正向完成率直接衡量"价值兑现"这一事件。不像"邮箱留存率"——邮箱是一个二阶的长期承诺信号，在产品还没证明单次价值之前就拿它做北极星，等于在没有短期价值的基础上追求长期信任，属于选错了优化对象。

**基准**：
- MVP 冷启动期 ≥ 50%（至少过半主动完成循环的用户在反馈中说"好一点"）
- 读数前置条件：分母至少达到 10 个完成循环的 session；低于此样本量时 rate 不稳定，仅观察不结论。

#### Phase 5+ 北极星升级：邮箱留存率

定义：解码完成后主动留邮箱的用户占比。

**何时升级**：当正向完成率稳定 ≥ 65% 且持续 3 周以上，意味着"单次价值"命题已验证；此时优化重心转向"长期信任 + 复利"，北极星切换到邮箱留存率。

**基准**：成熟期 ≥ 35%。

**分阶段北极星的意义**：错的阶段用错的指标会让团队优化错的事情。MVP 阶段追求"长期承诺"指标会诱导产品过早加复杂留存机制；成熟期还用"单次价值"指标则会让团队停留在早期优化陷阱。

### 6.2 健康度指标

| 指标 | 定义 | 健康阈值 |
|---|---|---|
| 邮箱留存率 | 解码完成后主动留邮箱的占比 | MVP ≥ 15%（作为未来北极星的早期读数） |
| 解码完成率 | landing → 看到解码卡 | ≥ 60% |
| 主按钮点击率 | 卡片 → 点「现在做 5 分钟」 | ≥ 70% |
| 回来率 | 点主按钮 → 回到二次触点 | ≥ 40% |
| 事后验证回复率 | 邮件点开 + 答题 | ≥ 30% |
| 7 日留存 | 首次解码后 7 天内再解码 | ≥ 25% |
| 手动修正分类比例 | 移动过 worry 的 session 占比 | < 20% 健康（高说明 AI 分类差） |

### 6.3 体验定性指标

- **完整 emoji 分布**（🙂 / 😐 / 😣 占比）—— 北极星之外的完整情绪光谱，不只看 🙂
- **"灾难化未发生" 占比** —— 产品核心命题的数据验证（洞察里的 70% 是否在真实数据里成立）
- 用户一句话反馈文本（情感方向，不做 NLP 评分）

- 多语言（中文优先；英文界面放上线后第一批）
- OG 图后端渲染（客户端截图够）

| 分享卡导出 | 客户端 html-to-image | MVP 不做后端 OG 图渲染 |

### 8.2 AI 管线演化叙事

#### v0 设计（6 次调用）

第一版按"A 类结构化 / B 类温度敏感"路由到 DeepSeek + Claude 双供应商。

```
v0 关键路径：
Call1 DeepSeek 1.5s → Call2 Haiku 3s → [用户] → Call3a DeepSeek 1.5s
  → Call3b Haiku 3s → [用户] → Call4a DeepSeek 1.5s → Call4b Sonnet 5s
  → Call5 Haiku 2s
关键路径：12-15s 累计 AI 等待
```

对焦虑型用户而言，等待时间本身就在加剧焦虑 —— 关键路径每多 1s 都是产品体验的负债。

#### v1 修订原则

从"按可见性 + 温度敏感性 + 结构化需求三维度路由"，收敛到"按任务风险分级路由 Haiku vs Sonnet"。供应商收敛到 Anthropic 单一家。

#### v1 调用链（5 次 AI 调用 + 1 个规则节点）

| Call | 任务 | 模型 | 关键路径？ | 说明 |
|---|---|---|---|---|
| 1 | detectAndAskFirst 状态识别 + 第一轮追问 | Haiku 4.5 | 是 | 结构化输出 `{state, question}`，question 部分 stream |
| — | needMoreInfo 判断（规则） | 无 AI | 是 | 字数 + 关键词启发式 |
| 2 | askFollowUpAgain 第二轮追问 | Haiku 4.5 | 仅当规则触发 | 硬上限 2 轮 |
| 3 | classifyAndCompose 切三堆 + 动作 + 标题 | Sonnet 4.5 tool use | 是 | 一次输出 `{worries, primary_action, headline}` |
| 4 | generateCompanionCopy 陪伴文案 | Haiku 4.5 | 否（按主按钮后） | 不在解码关键路径上 |
| 5 | composeVerificationEmail 验证邮件 | Haiku 4.5 | 否（异步 Cron） | 不在解码关键路径上 |

**优化后关键路径延迟**：
- 最短：Call 1 (4s) + Call 3 (5s) = ~9s，被用户输入打断成两段
- 最长：Call 1 (4s) + Call 2 (3s) + Call 3 (5s) = ~12s，被用户输入打断成三段
- 用户单次连续等待最长 ≤ 5s（解码揭开时刻）

#### 两处刻意的合并决策

**Call 1 把"识别状态"和"生成第一句追问"合并**：状态识别如果先单独调一次再传给追问 prompt，会出现"两种口吻在同一 session 内冲突"的风险，因为追问的语气本来就要适配状态。一次 Haiku 结构化输出 `{state, question}` 是更紧的耦合 —— 紧耦合在这里是优点不是缺点。

**Call 3 把"分类"和"生成动作 / 标题"合并到一次 Sonnet 调用**：如果先用便宜模型分类、再用 Sonnet 生成动作，Sonnet 看到的是"分类后的 paraphrase"而不是用户原话，会丢失对生成动作至关重要的微妙细节（用户用的具体动词、自我描述的情绪强度）。让 Sonnet 一次看到全部上下文的价值 > 省一次便宜调用的成本。

#### needMoreInfo 规则替代 AI 调用

以下任一条件命中 → 触发 Call 2 第二轮追问；都不命中 → 直接进 Call 3：

- 字数过少：用户首轮回答 < 50 字（实现时根据冷启动数据放宽到 30 字）
- 缺任务锚点：回答中无任何任务 / 动作动词
- 情绪纯描述：回答全是情绪词无具体事件

为什么用规则不用 AI：判断本身简单、规则 0ms 而 AI 调用 500ms+、规则可观察可调整不漂移。如果在 Phase 4 打磨阶段被证明效果不理想，可以升级为轻量 AI 调用，但 MVP 不在这里花预算。

### 8.3 代码组织与 iOS 预留

```
/app                         Next.js App Router（UI 层 — iOS 重写这一层）
  /(decode)/...
  /history/...
  /card/[sessionId]/...
  /verify/[token]/...
  /api/...                   薄壳：校验 + 调 /lib/core + 返回 JSON

/lib
  /core                      纯业务逻辑（无 UI / 框架依赖 — iOS 直接复用）
    sessionService.ts
    decodeEngine.ts
    quoteSelector.ts
  /prompts                   Prompt 模板（独立文件 — iOS 复用）
  /rules                     非 AI 的轻判断
  /ai                        LLM 客户端封装
  /db                        Supabase 数据访问

/data                        静态内容（构建时打包）
  openingQuotes.ts

/components                  React 组件（仅 Web）
```

**架构边界**：`/app/api/*` 路由是薄壳，业务逻辑全在 `/lib/core`。未来 iOS 接任意 BFF（FastAPI / Hono）都能直接复用 `/lib/core` 和 `/lib/prompts`。

这条物理隔离不是过度设计，是为了支持产品路线图（Phase 7 = iOS 客户端）而提前画的线。先用 Web 验证产品命题；命题成立后 iOS 重写 UI 层，业务逻辑零迁移成本。

### 8.4 隐私架构作为第三支柱

焦虑解码器收集的是全互联网最敏感的一类数据：用户最深层的焦虑、对自己的怀疑、对他人的恐惧。如果不从第一天就建立清晰的信任架构，产品会在两个维度同时失败：伦理失败（用户不知不觉把私密内容托付给不值得的系统）+ 增长失败（焦虑型用户对"这个 App 能看到我有多脆弱"极度敏感，信任缺失会在 landing 页就劝退）。

四层信任架构：

1. **数据最小化**：`initial_dump` 原文 30 天 TTL 自动删除；衍生分类长期保留；草稿只存 localStorage 永不上传
2. **API 层 Anthropic 默认不训练**：单一供应商（敏感数据不发往政策不清晰的厂商）
3. **`/settings/data` 数据驾驶舱**：查看 / 导出 / 分粒度删除
4. **Landing 隐私承诺一句话** + `/privacy` 详细说明页

隐私不是合规任务，是 **产品的第三支柱**（前两根：对焦虑型用户的洞察 / AI 工程成熟度）。

### 8.5 Prompt 调优作为持续工程

LLM 输出在"模型 + prompt + 用户输入"三个变量上漂移，无法靠传统单测断言。MVP 阶段建立**持续 benchmark + 人眼 diff** 的轻量工程，避免"prompt 改一次问题留一年"。

**Benchmark dataset（8 样本）**：覆盖矩阵 启动状态 / 救援状态 × 任务清晰 / 任务模糊 × 长 / 短，外加两条边缘 case（纯情绪无任务、纯任务无担心）。每次 prompt 改动跑一次完整 benchmark，输出 markdown 文件按时间戳保存到 `tests/prompts/benchmarks/runs/`，新旧版本人眼 side-by-side 对比。

**v1 → v2 实际调优 case**：v1 跑出来在两个失败模式上一致：(1) "课上不进去 / 项目推不动" 这种**状态描述**被错放进 real（应是 fog —— real 应该是名词性具体物件）；(2) "今天又过去了 / 我又拖了" 这种**自我评价**被装进 fog（污染 fog 类，应直接丢弃）。v2 在 prompt 里显式定义了"real = 名词性具体物件" + "自我评价 / 情绪 commentary 不归入任何 bucket，直接丢弃"。

**needMoreInfo 用规则不用 AI**：判断"用户回复是否够 AI 分类"是简单逻辑（字数 + 任务动词 + 情绪占比启发式）。规则 0ms 而 AI 调用 500ms+；规则可观察可调整不漂移。这是"哪些工作不应该交给 LLM"的实操样本。

**手动修正率 = AI 分类质量的反指标**：用户每次拖 worry 到另一堆，写入 `was_manually_edited`。某周修正率 > 20% 即 prompt 复盘信号。这是**用户行为驱动的免费质量治理**，不需要外部标注、不需要自动断言。

## 9. 风险与验证

### 9.1 风险矩阵

| 风险 | 概率 | 影响 | 缓释 |
|---|---|---|---|
| AI 分类质量不稳 | 中 | 高 | 5-8 个 benchmark 用例做回归；手动修正率作为兜底信号 |

| 面试官 demo 不愿写真担心 | 高 | 低 | 准备 2-3 条"示例焦虑"一键填充 |

| AI 成本失控 | 低 | 中 | IP 限流 + session 限频；关键路径大部分是 Haiku（便宜），仅 Call 3 用 Sonnet |
| 三方免费额度耗尽 | 低 | 低 | MVP 阶段够用，到上限再付费升级 |
| 事后验证邮件措辞激活焦虑 | 中 | 高 | 至少 3 版文案 + 真人测读（Phase 4 框架任务 F3） |

### 9.2 数据命题的证伪条件

产品的核心命题 —— **"约 70% 的任务回避，背后什么具体的事情都没有"** —— 必须可证伪。如果命题不成立，产品的设计基础就垮了，而不是简单地"AI 调一调"。

证伪条件（达到任一即视为命题被驳）：

- 累积 100 次解码后，fog 类 worry 占比稳定 < 30%（用户大多数担心都是真问题，不是雾）
- catastrophic 类 worry 的"未发生"比例稳定 < 50%（灾难化想象大多发生了，不是想象）
- 修正率长期 > 30%（用户系统性不认同 AI 的分类）

如果证伪：要回到"用户的焦虑形态是什么"重新做用户研究，而不是迭代当前产品。

如果证实：产品的差异化壁垒（一份属于自己的"灾难没发生"证据库）就是真实的复利价值。

### 9.3 体验定性指标作为辅助

- 完整 emoji 分布（🙂 / 😐 / 😣 占比）：北极星之外的完整情绪光谱
- 用户一句话反馈文本：情感方向，不做 NLP 评分
- 7 日留存：MVP 期 ≥ 25% 健康

## 10. 冷启动计划

### 10.1 为什么加这一节

"做了但没人用"是作品集项目的致命失败模式。一份不回答"谁会用"的 PRD 在面试中会被直接质疑 go-to-market 思维。这一节回答：MVP 上线后的前 4 周，如何获得 20-30 个真实用户和 50-80 次真实解码。

### 10.2 冷启动目标（MVP 上线后 4 周内）

| 指标 | 目标 | 层级 | 说明 |
|---|---|---|---|
| 真实用户数 | 20-30 | 规模 | 非自己、非重复设备 |
| 总解码次数 | 50-80 | 规模 | 允许大部分用户只试 1-2 次 |
| 完成完整循环的 session 数 | ≥ 10 | 北极星分母 | 走完 landing → 解码 → 主按钮 → 回来 → 反馈 |
| 正向完成率（🙂 占比） | ≥ 50% | 北极星早期读数 | MVP 阶段北极星 |
| 2 次及以上解码的用户 | ≥ 5 | 价值信号 | "产品真有价值"的硬信号 |
| 留邮箱用户 | ≥ 6 | 次级 | 未来北极星（邮箱留存率）的早期读数 |
| 已回答的事后验证 | ≥ 3 | 长期循环 | 长期闭环的最小可观察数据 |

**关于目标设定的诚实**：

- 这些数字低于常规产品启动目标，但对一份"PM 作品集 + 可用 demo"的验证已足够。
- 真正需要证明的不是"我能做出一个有人用的产品"，而是"我能让陌生人在没有我在旁边时，完成一次有价值的体验，并且至少过半说'帮到我了'"。≥ 10 个完整循环 × ≥ 50% 🙂 = 5 个 🙂 信号，是这个命题的最小证明单元。
- 样本量不大的诚实承认：10 个完整循环的 🙂 比例统计上仍有噪声。MVP 北极星是"早期读数"而不是"统计显著结论"，到 Phase 5+ 上规模后再追求显著性。

### 10.3 三层分发渠道（按信任层级从高到低）

#### Tier 1 · 密友种子

- 目标：10-15 人
- 来源：直接联系已知有焦虑 / 拖延困扰的朋友、同学、群友
- 方式：私信 + 一句话邀请 + demo 链接 + "能不能体验后发一段 30 秒语音告诉我哪里不顺"
- 作用：Phase 4 打磨的关键信号源

#### Tier 2 · 自有社交

- 目标：5-10 人转化
- 来源：小红书 / 即刻 / Twitter / 朋友圈的自有关注者
- 方式：发一篇围绕产品洞察的内容（不是推广文案）
- 作用：既是冷启动渠道，也是个人品牌沉淀

#### Tier 3 · 细分社区

- 目标：5-10 人转化
- 来源：知乎"如何克服拖延"、豆瓣拖延小组、Reddit r/GetMotivated / r/productivity
- 方式：诚恳分享 + 征集反馈（不是硬推广）
- 作用：最广但最不可控，用来补充总量

### 10.4 激活漏斗

```
点击链接进 landing      = 100%
  ↓ 首句提交率          ≥ 60%（landing 文案要让人敢写第一句）
进入对话                = 60%
  ↓ 看到解码卡率        ≥ 90%（AI 链路稳定性）
看到解码卡              = 54%
  ↓ 点主按钮率          ≥ 70%
点了"开始做 5 分钟"      = 38%
  ↓ 回来率              ≥ 40%
填完二次触点            = 15%  ← 完整循环完成率
  ↓ 🙂 反馈率           ≥ 50%  ← 北极星
🙂 反馈占首次访问        = 7.5% ← 产品真正"帮到"的转化率
```

漏斗任一环显著低于预期 → Phase 4 重点打磨那一环。最敏感的两环：

- **首句提交率（60%）**：决定漏斗入口宽度，与 landing 文案强相关
- **🙂 反馈率（50%）**：决定北极星读数，与解码卡质量 + 主按钮的 action 颗粒度强相关

### 10.5 关于种子用户的伦理边界

这个产品处理的是敏感话题。邀请朋友用 MVP 时：

- **不追问结果**：你邀请了他用，他用了没告诉你 → 不主动追问。他们用产品时的焦虑内容不是你的谈资。
- **不看后台数据定位个人**：即使有 DB 访问权，不要反查"我朋友小 A 的解码内容是什么"。
- **Tier 1 邀请时明示**：你是产品作者，但你看不到他们的具体内容。

这条写进 PRD 的同时也是个人价值观的表达。

## 11. 视觉系统（v1.0）

### 11.1 视觉策略：克制底色 + 暖色重音

视觉定位与产品定位（§3 "克制工具"）形成对应关系：UI 整体走**低饱和莫兰迪豆绿**底，让产品本身保留"工具"的克制感；而**手绘蜡笔风暖色猫图**作为情感重音落在底色上 —— 像白墙挂彩画的逻辑，重音不喧宾。

**为什么不走纯极简（如 Linear 的全白底）**：纯工具感的视觉对焦虑型用户有"被审问"的隐患。视觉里要有一点温度，但不能多到滑向治愈系。莫兰迪豆绿 + 单一手绘元素（猫）是这条窄路的具体表达。

**为什么不走纯温暖系（如 Headspace 的渐变橙紫）**：治愈系 App 的视觉一旦套用，产品命题（生产力工具不是治疗）会被一秒出卖。`spec §1.4` 反复警告这个陷阱 —— 视觉是 PRD 的最后一关。

### 11.2 色板

```
--bg              #DCE3CB    豆绿底（低饱和莫兰迪）
--text            #2E3A35    深灰带绿调
--text-muted      #6E7A6F    中灰
--card-bg         #FFFFFF    纯白卡（在豆绿底上的对比锚）
--card-border     #C4CAA9    沙绿边框
--input-border-focus  #4A6B5C  深沙绿（焦点态）

--bucket-real         #4A6B5C   深沙绿（任务）
--bucket-catastrophic #A4796E   暮色橘（灾难化）
--bucket-fog          #B5B2AC   暖灰（雾）

--accent          #6B8F82   muted teal（主按钮）
--sparkle         #E5C77F   暖金（魔法棒星星）
```

三色 bucket 的取色逻辑：real 用最深沉、最有"ground"感的沙绿；catastrophic 用暮色橘（既有"警示"又不像红色那样冒犯）；fog 用最弱的暖灰（视觉权重最低，对应 §5.4 "偏心第一堆"）。

### 11.3 字体系统

| 用途 | 字体 | 来源 |
|---|---|---|
| 中文标题 + 中文正文 | LXGW WenKai Lite（霞鹜文楷） | jsdelivr CDN，~3MB（首次载入慢，后续缓存） |
| 英文 / 数字装饰 | Caveat（Google Fonts） | next/font/google |

**用手写体作 body 字体反直觉**（多数产品建议正文用无衬线打字体）。但产品的核心情感诉求是"温柔陪伴"，字体本身就是最便宜的温度来源。代价是首次载入 ~3MB 字体文件 —— 用 `font-display: swap` 让首屏不阻塞，字体加载完再无感切换。

**为什么不用更小的中文手写字体（如 Long Cang / Ma Shan Zheng）**：那些字体太"夸张装饰"，行书 / 草书感强，不适合长段落的正文阅读。LXGW WenKai 是"半楷"，手写感存在但保留可读性，是温度与功能的平衡点。

### 11.4 6 张手绘猫图（Midjourney + niji 6）

每张猫图对应一个产品场景：

| 文件 | 场景 | 内容 |
|---|---|---|
| `01-opening-ritual.png` | Landing 仪式 | 黑白猫安静盘坐 |
| `02-decode-card.png` | 解码完成 | 白猫专注看三张色卡 |
| `03-wait.png` | 等候页 | 蓬松橘猫坐书桌前 |
| `04-return.png` | 我回来了 | 橘猫舒展配蓝茶杯 |
| `05-history.png` | 记忆库 | 白猫看玻璃罐 |
| `06-not-found.png` | 404 | 黑猫望云 |

**生成方法**：Midjourney `--sref <numeric_code> --sw 1000 --niji 6`。`--sref` 锁风格、`--sw 1000` 顶满 style weight、`--niji 6` 用专门的手绘动漫模型。第一张确定风格后，后续 5 张沿用同一 sref 保持画风一致。

**为什么不用素材库（unDraw / Storyset）**：素材库插图是"设计师的视觉假设" —— 平面、几何、抽象。蜡笔猫是"个人化的视觉符号"，每张都有手绘的不规则、笔触留白、温度。两者对"产品是个有温度的工具"的传达力差很多。

**插图作为视觉重音，不抢戏**：每张都用 `w-56` 到 `w-64`（224-256px）小尺寸，居中或顶部留白，不超过页面 1/3 的视觉权重。

### 11.5 MagicCursor 魔法棒 + SparkleBurst

桌面用户的鼠标变成手绘魔法棒（沙绿星 + 深棕杆 SVG），点击任意位置触发 8 颗暖金色星星向外炸开（framer-motion，~650ms，hotspot 落在星星中心而非杆尾）。

移动设备没有 cursor 概念，自动 fallback 到"重要交互才触发"模式：DecodeCard 的「现在做 5 分钟」、EmailOptIn 的提交、Return 的 emoji 提交，三处用 `useSparkleOnMobile` hook 在触摸点触发星星。

**为什么这是设计而不是装饰**：焦虑型用户的内心隐喻是"挤压、卡住、灰色"。给 cursor 加魔法棒 + 点击有星星，是把"轻松、玩耍、被祝福"的微感受嵌入每一个原本机械的交互。每次点按都是一个微小的"被庆祝"。

**为什么不让所有交互都触发（移动端）**：移动端如果每次点按都炸星星会很吵 —— 失去"重要时刻被庆祝"的仪式感，变成噪音。只在"完成一个产品意图"的按钮上触发，是把仪式感保留在需要的瞬间。

### 11.6 FlipCard 解码 ceremony

解码结果页（`/decode/:id/result`）的 DecodeCard 用一个 3D 翻面卡片包装：

- **正面**：白卡 + 居中的 `02-decode-card.png` 白猫看色卡 + 底部小字 "点一下看你的解码 ↻"
- **反面**：DecodeCard（worry 三堆 + 主按钮）
- **进入**：默认显示正面，1.5s 后**自动翻转**，spring 动画 750ms
- **手动翻**：点正面任意位置翻到反面；反面右上角有 ↻ 链接翻回正面
- **DecodeCard 内部交互不冲突**：CategoryChip 下拉、主按钮等内部点击不触发翻面（child stop propagation）

**为什么是 ceremony 而不是直接展示**：原版直接渲染 DecodeCard 时，用户进页面瞬间映入眼帘的是大卡片密密麻麻的内容，**视觉权重盖过了"啊，解码完成了"这件事本身**。FlipCard 让"解码完成"这一刻有一个 1.5 秒的**视觉停顿**：先看见猫（"它替你看完了"），再翻面看自己的解码（"现在轮到你"）。这是把功能事件转化为情感仪式的微设计 —— 与 §4.3 落地仪式的逻辑同源。

**用户可以反复翻**：翻回猫面再翻回卡面，是对"我可以暂停一下"的尊重。不是炫技，是给焦虑用户一个"喘口气"的物理动作。
