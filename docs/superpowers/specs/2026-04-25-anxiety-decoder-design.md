## 1. 产品概述

### 1.1 一句话定位

**焦虑解码器是面向学生与申请者的反内耗启动器** —— 帮你在"想做又动不了"的瞬间，5 分钟内戳穿模糊感、拿到一个能立刻动手的最小动作。

### 1.2 核心洞察

焦虑型拖延最骗人的地方，是它让你以为有一个具体的坏结果在等你。但把脑子里所有担心倒出来、戳穿之后，往往发现：

- 一部分是**真实需要处理的事**
- 一部分是**灾难化想象**
- 一部分是**根本说不清的雾**

**大约 70% 的任务回避，其实背后什么具体的事情都没有。** 产品的价值命题就建立在对这个现象的持续验证上 —— 用户每次使用完，都会积累一份属于自己的"灾难没发生"证据库。

### 1.3 核心用户

**学生 / 研究生 / 申请者。** 这类人群的任务形态匹配产品命题：

- 大块、无明确 deadline 的任务（论文、毕业设计、找工作、申请）
- 高频回避（没有同事/老板盯）
- 任务语言含糊（"投简历"、"准备面试"、"写文献综述"）
- 强灾难化想象（"投了会被拒"、"面试会表现很差"、"导师会觉得我不行"）

### 1.4 产品定位的边界

**不是**治疗 App、不是 CBT 工具、不是情绪日记、不是冥想陪伴。
**是**一款服务焦虑型使用者的**生产力工具**。

与三类邻居产品的差异：

| 对比对象 | 它们做什么 | 焦虑解码器做什么 |
|---|---|---|
| 待办类（Todoist / 滴答） | 假设你已知要做什么，帮你排序 | 服务"知道但启动不了"这一公里 |
| 冥想/CBT（Headspace / Woebot） | 让你"接受"焦虑 | 让你"绕过"焦虑去启动 |
| 日记/番茄（Daylio / Forest） | 记录心情或施压完成 | 只承担一件事：帮你开始下一步 |

## 3. 核心体验循环

### 3.1 单次解码闭环（一次使用内走完）

```
[落地仪式：随机一句反焦虑短句 1.5s 淡入淡出]   ← 见 §3.4
        ↓ 自动过渡（用户也可点击跳过）
[空白页 + 一段温柔文字 + 输入框]
        ↓ 用户写一段（自由文本）
[AI 自动识别：启动仪式 OR 救援模式]
        ↓ AI 追问 1~2 个针对性问题（硬上限 2 轮）
[解码卡生成]
   🟢 真问题 → 主视觉 + 「现在做 5 分钟」主按钮
   🟡 灾难化 → 静默入"事后验证队列"
   ⚪ 雾    → 静默存档
        ↓
[用户可手动修正分类 —— 拖/点将 worry 移到另一堆]
        ↓
[轻邮箱询问（可跳过）]"想让我 3 天后问问你那件事发生了吗？"
        ↓ 用户点主按钮
[等候页：陪伴文案 + 唯一一个「我回来了」按钮]
        ↓ 用户回来
[二次触点：刚才怎么样？emoji + 一句话]
        ↓
[历史卡片列表 + 长期证据数据]
```

### 3.2 长期价值循环（跨会话，异步）

```
解码时 catastrophic 类 worry 被标记
        ↓ 1-7 天后（用户选定）
[邮件触发 OR 站内兜底提醒]"当时担心的事发生了吗？"
        ↓ 用户回答
[单条 worry 标记为 did_happen: true/false]
        ↓ 累积
[个人证据库："你 30 次解码里，22 次灾难化想象最终没发生"]
```

### 3.3 关键设计决策（体验层）

1. **单入口，不让焦虑用户做选择题** —— 不分"启动模式"和"救援模式"两个入口；AI 从首句话自动识别，动态调整引导语气。
2. **「我回来了」必须用户主动点击** —— 不自动检测窗口焦点；让二次触点的触发显式化，避免用户关掉浏览器导致的价值流失。
3. **历史卡片是入口级一等公民** —— 长期证据数据是护城河，每次解码完都应让用户看到"过往 22 张卡里 16 张灾难没发生"。
4. **偏心第一堆** —— 解码卡的主视觉焦点只有一个："接下来要做什么"。其他两堆静默处理，价值在长期沉淀里。
5. **允许手动修正分类** —— "AI 是辅助"而非"AI 是权威"的姿态。同时作为 AI 质量的兜底信号（修正率 = 分类质量反指标）。
6. **加载时间装进仪式** —— 用户首屏的 1-2s 等待不是技术问题，是产品机会。落地仪式（§3.4）把不可避免的渲染时间变成减速带。

### 3.4 落地仪式（Opening Ritual）

#### 3.4.1 设计

每次用户访问 `/` 时，先呈现一句"反焦虑短句"，1.5s 淡入展示 + 0.5s 过渡，再自动接入输入表单。

**为什么必要**：
- 焦虑用户从其他页面切过来时，脑子是嗡嗡的；直接看到输入框反而会卡住"该写什么"。一句简短的、温柔的、非治疗式的短句，作为心理"减速带"，让用户从"日常焦虑回路"过渡到"愿意写下来"的状态。
- 这是把"必然的首屏渲染时间"变成产品价值的微设计 —— 加载页本来就是用户在等的 0.5-2s，把它装进仪式里，等待不再是损耗。

#### 3.4.2 呈现规范

| 维度 | 规范 |
|---|---|
| 时长 | 1.5s 显示 + 0.5s 淡出过渡到输入框，总计约 2s |
| 不阻塞 | 用户点击任意位置可立即跳过 |
| 视觉 | 纯净背景 + 单行短句居中，无装饰、无 logo、无加载圈 |
| 字号字体 | 与解码卡 `card_headline` 同款字体（视觉仪式感的一致性） |
| 减弱动效 | 命中 `prefers-reduced-motion: reduce` 时去掉淡入淡出，直接显示 + 缩短到 1s |

#### 3.4.3 短句内容原则

- **≤ 20 个汉字**
- **工具语气，不是疗愈语气**：观察句、锚定句优先；劝告句（"你应该"）禁用
- **不出现疗愈话术词汇**："焦虑""治愈""放松""平静""加油""勇敢"等禁用
- **留白感**：让读者自己接话，不替读者总结

#### 3.4.4 MVP 短句库（v1，10 条种子）

存储在 `/data/openingQuotes.ts` 静态文件：

```ts
export const openingQuotes = [
  "不去想两个小时以后和五公里外的事",
  "你脑子里 90% 的事此刻都不需要解决",
  "现在不知道答案是允许的",
  "只决定接下来五分钟",
  "焦虑常常是脑子在做没人要求它做的工作",
  "想得越远，能动的越少",
  "把一件事做完不是仪式，开始才是",
  "你不需要先平静下来才能开始",
  "脑子比你以为的可信度低",
  "下一步比下个月更重要",
];
```

#### 3.4.5 抽样规则

- 纯随机，但**避免最近 3 次显示过的**（用 localStorage 记录最近 3 条 index）
- 全部用完前不重复，重置滚动
- 抽样逻辑放在 `/lib/core/quoteSelector.ts`（纯函数，便于单测）

#### 3.4.6 何时不显示

- **MVP 阶段**：每次访问都显示。不做"近期访问跳过"逻辑，避免过早 over-engineering。
- 如果种子用户反馈"每次都看烦了"（§19 冷启动定性反馈），Phase 4 加"15 分钟内重访不显示"逻辑。

#### 3.4.7 V2+ 扩展方向

- 用户自己投稿短句（社区共建）
- 根据用户历史 catastrophic 频率个性化推送
- 时段适配（早晨 vs 深夜不同短句池）

## 4. 信息架构

### 4.1 路由

| 路径 | 角色 |
|---|---|
| `/` | Landing — 空白 + 引导文 + 输入框（主入口） |
| `/decode/[sessionId]` | 对话阶段（AI 追问） |
| `/decode/[sessionId]/result` | 解码结果（卡 + 主按钮） |
| `/decode/[sessionId]/wait` | 等候页（陪伴文案 + 「我回来了」） |
| `/decode/[sessionId]/return` | 二次触点（刚才怎么样） |
| `/history` | 历史卡片列表 + 长期证据数据 |
| `/card/[sessionId]` | 单卡详情页（可分享 / 截图） |
| `/verify/[token]` | 邮件点回的事后验证页 |

设计注：`/decode/:sessionId/*` 是同一会话的状态推进，用 Next.js nested route 支持深链恢复；`/history`、`/card/:id`、`/verify/:token` 是真正可外链的页面。

### 4.2 Session 状态机

```
draft → conversing → decoded → launched → returned → completed
                                    ↓
                             （有 catastrophic item 的）
                                verification 独立生命周期（异步）
                                    ↓
                              pending → sent → responded
```

## 5. 数据模型

### 5.1 核心表结构（Supabase / Postgres）

```
anonymous_users
  id              uuid PK
  fingerprint     text         浏览器指纹，localStorage 同步
  email           text NULL    用户主动留下（commitment device）
  created_at      timestamp

decode_sessions
  id              uuid PK
  user_id         uuid FK → anonymous_users
  state           enum         starting | rescue（AI 第一轮识别）
  initial_dump    text         首次输入
  conversation    jsonb        [{role, content, ts}, ...]
  primary_action  text         AI 给出的 5 分钟动作
  card_headline   text         卡片上的一句话标题
  status          enum         draft | conversing | decoded | launched | returned | completed
  created_at      timestamp
  updated_at      timestamp

worry_items                    一个 session 拆出多条，每条单独跟踪
  id              uuid PK
  session_id      uuid FK
  content         text         单条担心原话
  category        enum         real | catastrophic | fog
  display_order   int
  was_manually_edited  bool    用户是否手动修改过分类（质量埋点）

verifications                  仅 catastrophic 类 worry 生成
  id              uuid PK
  worry_item_id   uuid FK → worry_items
  scheduled_for   timestamp    用户选的天数
  sent_at         timestamp NULL
  token           text UNIQUE  邮件点回用
  did_happen      boolean NULL 用户回答
  user_note       text NULL
  responded_at    timestamp NULL

return_feedback                二次触点轻反馈
  session_id      uuid PK & FK
  emoji           text         🙂 | 😐 | 😣
  one_liner       text NULL
  created_at      timestamp
```

### 5.2 数据模型决策

1. **WorryItem 一等公民** —— 不塞进 session jsonb，因为事后验证按单条颗粒发送，累积统计要好做。
2. **Session status 与 verification 独立** —— verification 异步，不能阻塞主流程闭环。
3. **fingerprint 而非账号** —— 零门槛入口的实现路径；换设备会丢历史，MVP 接受。
4. **was_manually_edited 字段** —— AI 分类质量的核心反馈信号，用于长期 prompt 调优。

## 7. 技术架构

### 7.1 技术栈

| 维度 | 选型 | 理由 |
|---|---|---|
| 前端框架 | Next.js 15 App Router + TypeScript | 作品集标配，Vercel 一键部署 |
| 样式 | Tailwind CSS | 快速迭代卡片视觉 |
| 数据库 | Supabase (Postgres) | Free tier 够 MVP，RLS 为未来账号铺路 |
| AI 供应商 | **仅 Anthropic Claude**（Haiku 4.5 + Sonnet 4.5） | 见 §7.2 架构演化说明 |
| 邮件 | Resend | DX 好、3000 封/月免费 |
| 定时任务 | Vercel Cron | 每小时扫 verifications 表触发邮件 |
| 身份 | `crypto.randomUUID()` + localStorage | 零门槛；不做真指纹（隐私友好） |
| 分享卡导出 | 客户端 html-to-image | MVP 不做后端 OG 图渲染 |

### 7.2 AI 管线（按风险分级路由 Haiku/Sonnet）

#### 7.2.1 架构演化说明

第一版设计中，我提出了"按 A 类（结构化）/ B 类（温度敏感）路由 DeepSeek + Claude 双供应商"的架构。在实现规划阶段，我对完整的关键路径做了延迟分析：

```
v0 设计（6 次 AI 调用）：
Call1 DeepSeek 1.5s → Call2 Haiku 3s → [用户] → Call3a DeepSeek 1.5s
  → Call3b Haiku 3s → [用户] → Call4a DeepSeek 1.5s → Call4b Sonnet 5s
  → Call5 Haiku 2s
关键路径：12-15s 累计 AI 等待
```

对**焦虑型用户**而言，等待时间本身就在加剧焦虑 —— 关键路径每多 1s 都是产品体验的负债。

**修订后的架构原则**：从"按可见性 + 温度敏感性 + 结构化需求三维度路由"，收敛到"按任务风险分级路由 Haiku（轻 / 短 / 可重做）vs Sonnet（高 stakes / 长 stakes / 用户反复看）"。供应商收敛到 Anthropic 单一家。

**架构叙事的演化本身比初版方案更值钱**：从"我应用了一个分类模式"升级为"**我设计了一个分类方案、测了关键路径延迟、发现它服务故事而非用户、收敛到更简单的版本**"。

#### 7.2.2 修订后的调用链（5 次 AI 调用 + 1 个规则节点）

| Call | 任务 | 模型 | 关键路径？ | 说明 |
|---|---|---|---|---|
| 1 | detectAndAskFirst 状态识别 + 第一轮追问 | Haiku 4.5 | ✅ | 结构化输出 `{state, question}`，question 部分 stream |
| — | needMoreInfo 判断（规则） | 无 AI | ✅ | 字数 + 关键词启发式（见 7.2.3） |
| 2 | askFollowUpAgain 第二轮追问 | Haiku 4.5 | ✅ 仅当规则触发 | 硬上限 2 轮 |
| 3 | classifyAndCompose 切三堆 + 动作 + 标题 | Sonnet 4.5 tool use | ✅ | 一次输出 `{worries, primary_action, headline}` |
| 4 | generateCompanionCopy 陪伴文案 | Haiku 4.5 | ❌（按主按钮后） | 不在解码关键路径上 |
| 5 | composeVerificationEmail 验证邮件 | Haiku 4.5 | ❌（异步 Cron） | 不在解码关键路径上 |

**优化后关键路径延迟**：
- 最短：Call 1 (4s) + Call 3 (5s) = ~9s，被用户输入打断成两段
- 最长：Call 1 (4s) + Call 2 (3s) + Call 3 (5s) = ~12s，被用户输入打断成三段
- 用户单次连续等待最长 ≤ 5s（解码揭开时刻）

**两处刻意的合并决策**：

1. **Call 1 把"识别状态"和"生成第一句追问"合并** —— 状态识别如果先单独调一次再传给追问 prompt，会出现"两种口吻在同一 session 内冲突"的风险，因为追问的语气本来就要适配状态。一次 Haiku 结构化输出 `{state, question}` 是更紧的耦合 —— 紧耦合在这里是优点不是缺点。

2. **Call 3 把"分类"和"生成动作/标题"合并到一次 Sonnet 调用** —— 如果先用便宜模型分类、再用 Sonnet 生成动作，Sonnet 看到的是"分类后的 paraphrase"而不是用户原话，会丢失对生成动作至关重要的微妙细节（比如用户用的具体动词、自我描述的情绪强度）。Sonnet 做结构化分类的质量已经够，**让 Sonnet 一次看到全部上下文的价值 > 省一次便宜调用的成本**。

#### 7.2.3 needMoreInfo 规则（替代原 Call 3a 的 AI 调用）

以下任一条件命中 → 触发 Call 2 第二轮追问；都不命中 → 直接进 Call 3：

- **字数过少**：用户首轮回答 < 50 字
- **缺任务锚点**：用户回答中无任何任务/动作动词（正则 + 小词表："写 / 改 / 投 / 联系 / 打 / 见 / 读 / 复习 / 准备" 等）
- **情绪纯描述**：回答全是情绪词无具体事件（关键词："太累 / 难受 / 卷死了 / 不想动 / 烦"）

**为什么用规则不用 AI**：
- 判断本身简单（"信息够不够"是低复杂度任务）
- 用 AI 调用要 500ms+，规则 0ms
- 规则可观察、可调整、不漂移

如果规则在 Phase 4 打磨阶段被证明效果不理想，可以升级为轻量 AI 调用（加 Call 2.5），但 MVP 不应该在这里花预算。

#### 7.2.4 关键约束

- **追问硬上限 2 轮** —— 产品原则，不可配置。
- **分类步用 tool use 强制结构化** —— 不解析自由文本。
- **追问 stream，解码不 stream** —— 对话感 vs 仪式感的节奏区分。
- **有疑义时归入「雾」** —— prompt 中明确的优先级，避免 AI 自作主张硬塞。
- **`state` 作为下游 Call 的输入**：Call 1 识别出的 `state` 作为参数传入 Call 2（追问语气）、Call 3（分类 prompt 的语境提示）、Call 4（陪伴语气），保持一次会话内口吻连贯。

## 8. 关键设计原则 & 边界

| # | 原则 | 背后权衡 |
|---|---|---|
| 1 | 单入口，不做模式选择题 | 焦虑用户最怕选择题；AI 识别代替 UI 选择 |
| 2 | 追问硬上限 2 轮 | 不对焦虑用户做无限审问；够不够都停 |
| 3 | 偏心第一堆 | 用户离开时只记住一件事："接下来做什么" |
| 4 | 允许手动修正分类 | AI 是辅助不是权威；修正率作为质量反指标 |
| 5 | 「我回来了」主动点击 | 二次触点必须显式触发 |
| 6 | 不做计时器 | 产品承诺是"帮你开始"，不是"监督你做完" |
| 7 | 邮箱作为 commitment device | 留邮箱 = 对未来的自己多一层问责 |
| 8 | 分类有疑义时倾向「雾」 | 避免"AI 硬塞"的挫败感 |

## 9. 错误处理 & 边界情况

| 场景 | 处理 |
|---|---|
| LLM 调用失败 | 重试 1 次指数退避 → 失败则「AI 走神了，再来一次？」+ 重试按钮。**不降级规则分类** |
| 用户中途关闭浏览器 | 输入 debounce 每 2s 写 localStorage；session 创建后 id 存 localStorage；刷新可回到对话状态 |
| 用户拒绝留邮箱 | catastrophic item 仍入库 + 调度验证，但不发邮件；下次打开弹"还有 N 条待验证" |
| Prompt injection | system prompt 永远在 user 内容前；user 内容 XML 包裹；不做过度过滤 |
| 换设备 / 清缓存 | MVP 接受丢历史；V2 加 magic link 邮箱登录恢复 |
| Supabase / Resend 免费额度耗尽 | MVP 阶段 3000 封/月够用；上限后付费升级 |

## 12. 指标体系

### 12.1 北极星（分阶段）

**MVP 阶段北极星：正向完成率（Positive Cycle Rate）**

**定义**：所有进入二次触点（已点击"我回来了"并填完反馈）的 session 中，情绪反馈为 🙂 的比例。

**为什么是 MVP 阶段的北极星**：

MVP 阶段要验证的核心命题不是"用户会长期依赖这个产品"，而是"**用户单次使用完之后，是否真的感到帮助**"。

正向完成率直接衡量"价值兑现"这一事件。不像"邮箱留存率"——邮箱是一个二阶的长期承诺信号，在产品还没证明单次价值之前就拿它做北极星，等于在没有短期价值的基础上追求长期信任，属于选错了优化对象。

**基准**：
- MVP 冷启动期 **≥50%**（至少过半主动完成循环的用户在反馈中说"好一点"）
- 读数前置条件：分母至少达到 10 个完成循环的 session；低于此样本量时，rate 不稳定，仅观察不结论。

---

**Phase 5+（产品成熟期）北极星将升级为：邮箱留存率**

定义不变：解码完成后主动留邮箱的用户占比。

**何时升级**：当正向完成率稳定 ≥65% 且持续 3 周以上，意味着"单次价值"命题已验证；此时优化重心转向"长期信任 + 复利"，北极星切换到邮箱留存率。

**基准**：成熟期 ≥35%。

**分阶段北极星的意义**：错的阶段用错的指标会让团队优化错的事情。MVP 阶段追求"长期承诺"指标会诱导产品过早加复杂留存机制；成熟期还用"单次价值"指标则会让团队停留在早期优化陷阱。

### 12.2 健康度指标

| 指标 | 定义 | 健康阈值 |
|---|---|---|
| 邮箱留存率 | 解码完成后主动留邮箱的占比 | MVP ≥15%（作为未来北极星的早期读数） |
| 解码完成率 | landing → 看到解码卡 | ≥60% |
| 主按钮点击率 | 卡片 → 点「现在做 5 分钟」 | ≥70% |
| 回来率 | 点主按钮 → 回到二次触点 | ≥40% |
| 事后验证回复率 | 邮件点开 + 答题 | ≥30% |
| 7 日留存 | 首次解码后 7 天内再解码 | ≥25% |
| 手动修正分类比例 | 移动过 worry 的 session 占比 | **<20% 健康**（高说明 AI 分类差） |

### 12.3 体验定性指标

- **完整 emoji 分布**（🙂 / 😐 / 😣 占比） —— 北极星之外的完整情绪光谱，不只看 🙂
- **"灾难化未发生" 占比** —— 产品核心命题的数据验证（洞察里的 "70%" 是否在真实数据里成立）
- 用户一句话反馈文本（情感方向，不做 NLP 评分）

## 13. 风险矩阵 & 缓释

| 风险 | 概率 | 影响 | 缓释 |
|---|---|---|---|
| AI 分类质量不稳 | 中 | 高 | 5-8 个 benchmark 用例做回归；手动修正率作为兜底信号 |
| 用户不留邮箱 | 高 | 中 | 站内兜底验证；鸡蛋不押一个篮子 |
| 面试官 demo 不愿写真担心 | 高 | 低 | 准备 2-3 条"示例焦虑"一键填充 |

| AI 成本失控 | 低 | 中 | IP 限流 + session 限频；关键路径大部分是 Haiku（便宜），仅 Call 3 用 Sonnet |
| 三方免费额度耗尽 | 低 | 低 | MVP 阶段够用，到上限再付费升级 |

## 14. 测试策略（MVP 级）

- **AI 管线的端到端样本测试** —— 5~8 个真实担心样本覆盖启动/救援 × 清晰/模糊 × 长/短；每次 prompt 改动做一次回归（人眼 diff，不做自动断言，因为 LLM 输出漂移让断言脆）。
- **业务逻辑层（/lib/core）单元测试** —— 状态机转移、verification 调度时间计算这些纯函数，Vitest 覆盖。
- **端到端关键路径** —— Playwright 跑一条「首次进入 → 解码 → 点主按钮 → 回来 → 反馈」，确保路由链不断。
- **不做**：UI 组件单测（性价比低）、prompt 的自动断言（漂移不可靠）。

## 18. 隐私与信任设计

### 18.1 为什么这是独立章节

焦虑解码器收集的是**全互联网最敏感的一类数据**：用户最深层的焦虑、对自己的怀疑、对他人的恐惧。如果不从第一天就建立清晰的信任架构，产品会在两个维度同时失败：

- **伦理失败** —— 用户不知不觉把私密内容托付给不值得的系统
- **增长失败** —— 焦虑型用户对"这个 App 能看到我有多脆弱"极度敏感，信任缺失会在 landing 页就劝退

隐私不是合规任务，是**产品的第三支柱**（前两根：对焦虑型用户的洞察 / AI 工程成熟度）。

### 18.2 四层信任架构

#### 18.2.1 数据最小化：能不存就不存

- `initial_dump` 原文字段设置 **30 天 TTL**，自动删除。Cron 每日扫描 `created_at < now - 30 days` 的 session，将 `initial_dump` 置空，但保留 `worry_items` 和 `classification`（这些是衍生的、脱敏的、对用户有长期价值的部分）。
- `conversation` jsonb 同步脱敏：用户原话保留 30 天，AI 追问保留（AI 回复本身不敏感）。
- Draft（用户输入过程中的草稿）**永不上传**，只存 localStorage。即使用户关了页面，草稿也只在本机。

#### 18.2.2 API 层面：向 Anthropic 关闭训练，不接入政策不清晰的厂商

- **Anthropic**：API 默认不用数据训模型（[官方政策](https://privacy.anthropic.com/en/articles/7996868-is-my-data-used-for-model-training)），无需额外 header。在代码注释中明确记录这条承诺。
- **DeepSeek（已弃用）**：原 v0 设计包含 DeepSeek，但其数据训练政策在 TOS 中表述模糊。v1 架构演化后（见 §7.2.1），DeepSeek 已完全退出，敏感数据不会发往政策不清晰的厂商。**这是 §7.2 架构简化的附带收益**。
- 未来若接入其他供应商（如 V2 考虑加的视觉生成），必须先完成"数据训练政策核实"→"显式 opt-out 机制"→"用户告知" 三步。

#### 18.2.3 用户可见的"数据驾驶舱"

`/settings/data` 页面（MVP 必做）：

- **查看**：这个浏览器下存储的所有数据一览（session 数、worry item 数、是否留了邮箱）
- **导出**：一键下载所有数据为 JSON（用户对自己数据的所有权）
- **删除**：分粒度
  - 删除单个 session
  - 删除所有 session 保留 email（保留订阅 verification 邮件）
  - 删除所有数据（调用 `/api/settings/purge`，清 localStorage + 服务端级联删除）
- **透明度**：页面顶部一句话 —— "我们存了什么、存多久、为什么存。"

#### 18.2.4 Landing 第一屏的信任承诺

在输入框下方放一行小字（非弹窗、非勾选框、不打断流程）：

> 你写的东西只用来帮你解码这一次。原文 30 天后自动删除。[了解更多 →]

"了解更多"跳 `/privacy`，一页能读完的隐私说明，不用法律体裁。

### 18.3 实现优先级

| 措施 | Phase | 理由 |
|---|---|---|
| Landing 隐私承诺一句话 | Phase 1 | 冷启动第一天就需要 |
| `/api/settings/purge` 端点 | Phase 1 | 已在 §10 MVP 清单 |
| 30 天 TTL 的 Cron 任务 | Phase 2 | 复用 Phase 2 的 Cron 基础设施 |
| `/settings/data` 数据驾驶舱页面 | Phase 3 | 长期价值层的一部分，与历史页协同 |
| `/privacy` 详细说明页 | Phase 4 | 打磨阶段写清楚，上线前完备 |

### 18.4 隐私作为作品集叙事

在 PRD 和面试中，这条叙事比"加密/合规"硬核得多：

> "我设计这个产品时，第一版架构用 DeepSeek 做分类调用 —— 直到我意识到我无法向用户证明 DeepSeek 不会用他们的焦虑内容训练模型。我宁可放弃一个工程故事，也不愿把用户最私密的话发给政策不清晰的厂商。架构简化的决定，一半是为了延迟，一半是为了信任。"

## 19. 冷启动与分发计划

### 19.1 为什么加这一节

"做了但没人用"是作品集项目的致命失败模式。一份不回答"谁会用"的 PRD 在面试中会被直接质疑 go-to-market 思维。这一节回答：**MVP 上线后的前 4 周，如何获得 20-30 个真实用户和 50-80 次真实解码。**

### 19.2 冷启动目标（MVP 上线后 4 周内）

| 指标 | 目标 | 层级 | 说明 |
|---|---|---|---|
| 真实用户数 | **20-30** | 规模 | 非自己、非重复设备 |
| 总解码次数 | **50-80** | 规模 | 允许大部分用户只试 1-2 次 |
| 完成完整循环的 session 数 | **≥ 10** | 北极星分母 | 走完 landing → 解码 → 主按钮 → 回来 → 反馈的 session 数；低于此样本量时北极星 rate 读数不稳定 |
| **正向完成率（🙂 占比）** | **≥ 50%** | **北极星早期读数** | §12.1 MVP 阶段北极星 |
| 2 次及以上解码的用户 | **≥ 5** | 价值信号 | "产品真有价值"的硬信号 |
| 留邮箱用户 | **≥ 6** | 次级 | 未来北极星（邮箱留存率）的早期读数 |
| 已回答的事后验证 | **≥ 3** | 长期循环 | 长期闭环的最小可观察数据 |

**关于目标设定的诚实**：
- 这些数字**低于常规产品启动目标**，但对一份"PM 作品集 + 可用 demo"的验证已足够。
- 真正需要证明的不是"我能做出一个有人用的产品"，而是"**我能让陌生人在没有我在旁边时，完成一次有价值的体验，并且至少过半说'帮到我了'**"。≥10 个完整循环 × ≥50% 🙂 = 5 个 🙂 信号，是这个命题的最小证明单元。
- 样本量不大的诚实承认：10 个完整循环的 🙂 比例统计上仍有噪声（单个用户的情绪波动就会抖动 10% 读数）。MVP 北极星是"早期读数"而不是"统计显著结论"，到 Phase 5+ 上规模后再追求显著性。

### 19.3 三层分发渠道（按信任层级从高到低）

#### Tier 1 · 密友种子（信任最高、规模最小、质量最可控）

- **目标**：10-15 人
- **来源**：直接联系已知有焦虑/拖延困扰的朋友、同学、群友
- **方式**：私信 + 一句话邀请 + demo 链接 + "能不能体验后发一段 30 秒语音告诉我哪里不顺"
- **作用**：Phase 1 验收后的第一批反馈，用于 Phase 4 打磨的关键信号源

#### Tier 2 · 自有社交（信任中、规模中、质量中）

- **目标**：5-10 人转化
- **来源**：小红书 / 即刻 / Twitter / 朋友圈的自有关注者
- **方式**：发一篇围绕产品洞察的内容（不是推广文案），比如"我发现 70% 的拖延背后，其实什么具体的事情都没有。做了个工具验证这件事：[link]"
- **作用**：既是冷启动渠道，也是个人品牌沉淀（可复用为面试谈资）

#### Tier 3 · 细分社区（信任低、规模大、质量不稳）

- **目标**：5-10 人转化
- **来源**：知乎"如何克服拖延"、豆瓣拖延小组、Reddit r/GetMotivated / r/productivity
- **方式**：诚恳分享 + 征集反馈（**不是硬推广**，社区会反感）
- **作用**：最广但最不可控，用来补充总量 + 测"陌生人冷启动"转化

### 19.4 激活漏斗的关键节点

从 Tier X 拉过来的用户 → landing 到"产品真的帮到他"的逐级转化：

```
点击链接进 landing      = 100%
  ↓ 首句提交率          ≥ 60%（关键：landing 文案要让人敢写第一句）
进入对话                = 60%
  ↓ 看到解码卡率        ≥ 90%（AI 链路稳定性 + 不 bug 的问题）
看到解码卡              = 54%
  ↓ 点主按钮率          ≥ 70%
点了"开始做 5 分钟"      = 38%
  ↓ 回来率              ≥ 40%
填完二次触点            = 15%  ← 完整循环完成率
  ↓ 🙂 反馈率           ≥ 50%  ← 北极星（§12.1）
🙂 反馈占首次访问        = 7.5% ← 产品真正"帮到"的转化率
```

**对冷启动规模的反推**：
- 30 Tier 1/2 用户 × 7.5% = **约 2 个 🙂 信号**（来自首次访问者）
- 但 50-80 次总解码里，回访用户的完整循环完成率会显著高于 15%（他们已熟悉流程），估算总 🙂 信号 **≈ 5-8 个**
- 这和 §19.2 的"≥5 个 🙂 信号"目标一致

漏斗任一环显著低于预期 → Phase 4 要重点打磨那一环。最敏感的两环：
- **首句提交率（60%）**：决定冷启动漏斗的入口宽度，与 landing 文案强相关
- **🙂 反馈率（50%）**：决定北极星读数，与解码卡质量 + 主按钮的 action 颗粒度强相关

### 19.5 关于种子用户的伦理边界

这个产品处理的是敏感话题。邀请朋友用 MVP 时：

- **不追问结果**：你邀请了他用，他用了没告诉你 → 不主动追问。他们用产品时的焦虑内容不是你的谈资。
- **不看后台数据定位个人**：即使你有 DB 访问权，不要反查"我朋友小 A 的解码内容是什么"。
- **Tier 1 邀请时明示**：你是产品作者，但你看不到他们的具体内容（这点在 §18 信任设计里已保证）。

这条写进 PRD 的同时也是**个人价值观的表达** —— 面试官会注意到。
